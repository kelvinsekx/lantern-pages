## Console.log out

`console.log` in node is a web syntatic wrapper. It is a convenience alternative to `process.stdout.write()` function.

The console is a **global instance** configured to write to node's standard out among other things. Because of being global, the `.log()` function can be used without importing it from `node:console`.

`console.log` is used for debugging or display a record to the client.

<AsideNote>
  {`If process.stdout, process.stderr confuse you, don't mind it. You will see
  more lengthier explanation on them later.`}
</AsideNote>

Let's run some examples in the REPL.

```bash

$ node
Welcome to Node.js v21.6.0.
Type ".help" for more information.
> console.log("kelvin")
kelvin
undefined
> process.stdout.write('kelvin \n')
kelvin
true
```

In subsequent examples and the rest of the course, I'll skip these terminal details and jump straight at the sample code with their expected outputs so that the example above is rewritten with fewer lines.

```bash

console.log("kelvin") // kelvin
process.stdout.write('kelvin \n') // kelvin
```

## Command arguments with process.argv

A Node program can read its command-line arguments from `process.argv`.
Process.argv is an array that contains the commands passed when the Node.js process was launched. The first element of this array will be the path to the Node executable, and whatever follows your node command.

Run this in a REPL

```bash
process.argv // [ '/path/to/yours/versions/node/v21.6.0/bin/node' ]
```

To explore an example, let's run a script.

For example, suppose you save this Node program to the file argv1.js:

```bash
//file: node-example-1/argv1.js
console.log(process.argv)
```

Run this command in the parent directory from where the argv1.js file is located.

```bash
$ node argv1.js

/**
[
  '/path/to/yours/versions/node/v21.6.0/bin/node',
  '/path/to/folder/argv1.js'
]
*/
```

<LongAside subject="A couple of things to keep in mind">

<ol>
  <li>
    By default when you run node via script method the second argument (argvs)
    is always the script name.
  </li>
  <li>
    The first and second elements of process.argv will be fully qualified
    filesystem paths to the Node executable and the file of JavaScript that is
    being executed.
  </li>
  <li>
    Command-line arguments that are intended for and interpreted by the Node
    executable itself are consumed by the Node executable and do not appear in
    process.argv.

    <LongAside subject="Arguments not in process.argv">
    Arguments intended for the Node runtime (and not your script) don't appear in the process.argv. Node parses and use them before loading the JavaScript file and do not need to pass them to your program.

    | Flag      | Purpose                                     |
    | --------- | ------------------------------------------- |
    | --inspect | Enable the debugger                         |
    | --require | preload a module before running your script |

    If you want to access all arguments including those used by node, you can use `process.execArgv`.
    </LongAside>

  </li>
</ol>

</LongAside>

## Environment variables with process.env

Node programs can also take input from Unix-style environment variables. These variables are available through **process.env**.

<LongAside subject="Unix-style environment variables?">
We could have written that sentence as: **_Node programs can also take input from environment variables_**. The adjective Unix-style is to emphasize the acceptable style when writing that specific style of variables, environment variables.

Some of the standard that guides this style are:

1. All character must be written in UPPERCASE. It's the name convention and the most visible characteristics of this style.
2. All values of a variable must be stored as strings (even numbers). No real data types like int, bool ecosystem
3. No space in variable assigment. i.e between the KEY and VALUE e.g `count = 42` is wrong; `count=2` is the right way.

</LongAside>

Process.env is an object, whose property are the environment variable names and their string values. It contains your computer's environment variables.

Process.env is important for its perfect use case as configurations or variables that might change between deployments.Hide sensitive information that would have otherwise be hardcoded.

Using the REPL, run `process.env`. Its output is a long object.

To get our hands dirty, create some custom variables in a `.env` file in the root directory where you are running your node applications.

```bash
$ vim .env
```

Paste the code below in the file

```bash

//file: .env
PORTAL=SEKX_10001
```

Exit the REPL if it was running while you made the addition to the env file and run the REPL again.

In the REPL run `process.env`. Try find your newly defined variables. Or better still do `process.env.PORTAL`.

Oops, you wouldn't be able to find it in the output or `process.env.PORTAL` should return `undefined`. This is because you would manually load the env file if you want the variables to be included in your system's environment.

Start your REPL with `node --env-file=.env`. It loads the .env file in your REPL environment. `process.env.PORTAL` would return "SEKX_10001". Celebrationssss 🍕

## Modules and npm (Node Package Manager)

Node had its module system before javascript's'. This system is called CommonJS.

The then Node's module system uses the `require()` function to import values into a module and the `exports` object or the `module.exports` property to export values from a module.

In javascript, a module is typically a file. Not all programming languages behave this way. For some, multiple modules can exist in a single file.

**What is a module?**

In recent versions, after JavaScript rolled out it own module system, we now have 2 combined module systems available for us. This later module system is called ES6 module, and the earlier one is Commonjs module. These modules are not compatible so you are forced to stick with one in your project. The recommended one is ES6 module except you are working in a legacy and large codebase written in CommonJS.

Common js

```bash
// calc.js
const add = function(a, b){
  return a  +  b;
}
exports.add

// main.js
const { add } = require('./calc.js');

console.log(add(2, 4));
```

ES6 Module

```bash
// calc.js
const add = function(a, b){
  return a  +  b;
}
export add;

// main.js
import { add } from './calc.js';

console.log(add(2, 4));
```

<LongAside subject="A little deep on module and extensions">
The simplest way to tell Node what kind of module it is loading is to encode this information in the file extension. If you save your JavaScript code in a file that ends with .mjs, then Node will always load it as an ES6 module, will expect it to use import and export, and will not provide a require() function. And if you save your code in a file that ends with .cjs, then Node will always treat it as a CommonJS module, will provide a require() function, and will throw a SyntaxError if you use import or export declarations.

For files that do not have an explicit .mjs or .cjs extension, Node looks into the package.json file in the same directory as the file. Once the nearest package.json file is found, Node checks for a top level type property in the JSON object. If the value of the type property is "module", then Node loads the file as an ES6 module. If the value of that property is "commonjs", then Node loads the files as a CommonJS module. Note that you do not need to have a package.json file to run Node programs: when no such file is found (or when the file is found but it does not have a type property), Node defaults to using CommonJS modules.

</LongAside>
## Node Manager Packager

On installing node, it is accompanied with the Node Package Manager, NPM. NPM is the official package manager for the JavaScript ecosystem.

NPM downloads and manage libraries you would be using in your code. It keep tracks of these dependencies (as well as other info about these dependencies) in files **package.json** and **package-lock.json** in _the root directory_.
